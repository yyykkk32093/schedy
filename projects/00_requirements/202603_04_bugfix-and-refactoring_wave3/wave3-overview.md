## ⚠️ 設計判断

### D-1: [10] アクティビティ詳細パス見直し — 見送り

**提案**: `/activities/:activityId?schedule=xxx` → `/communities/:communityId?activity=xxx&schedule=xxx`

**見送り理由**:
| 観点               | 評価                                                                                                       |
| ------------------ | ---------------------------------------------------------------------------------------------------------- |
| REST原則           | アクティビティは独立リソース（固有ID）→ クエリパラメータ化は設計の退化                                     |
| 責務分離           | CommunityDetailPage（タブ管理）にActivityDetailPage（660行）のロジック統合は画面肥大化                     |
| パラメータ衝突     | `/communities/:id` は既に `?tab=` を使用中。`?activity=&schedule=` との共存が複雑                          |
| マルチコンテキスト | BottomNav・ホームフィード・コミュニティ内タブ等、複数入口からアクセスされるため communityId 非依存が合理的 |
| ブラウザ履歴       | クエリパラメータのみ変更は戻る/進むの挙動が不自然になるリスク                                              |

**将来的にやるならネスト型が自然**:
`/communities/:communityId/activities/:activityId?schedule=xxx`
- アクティビティはコミュニティ従属リソース（Activity.communityId は必須FK）
- 一覧(`/communities/:id/activities`)・作成(`/communities/:id/activities/new`)は既にネスト型
- 詳細だけトップレベルなのは一貫性に欠ける
- ただし変更箇所8箇所+のコストがあるため、wave4でまとめてリファクタする

→ **wave4にネスト型リファクタとして移送**
