## W3-07 削除系整理（チャット退出・コミュニティ退出・ユーザー退会）

- **分類**: 改善
- **優先度**: P2
- **変更対象**: 両方
- **変更レイヤー**: UI / API / UseCase / Domain
- **由来**: 積み残し #3
- **依存**: なし

### 仕様
3パターンの退出/削除フローを整理し、統一的な確認UI + バックエンド処理を実装する。

#### ① チャット退出
| 項目             | 内容                                                                     |
| ---------------- | ------------------------------------------------------------------------ |
| トリガー         | チャット画面の「チャットを抜ける」ボタン                                 |
| 確認ダイアログ   | 「このチャットから退出しますか？メッセージ履歴は閲覧できなくなります。」 |
| バックエンド処理 | DMParticipant レコードを削除（物理削除）                                 |
| 影響範囲         | チャット一覧から該当チャンネルが消える                                   |

#### ② コミュニティ退出
| 項目             | 内容                                                                                                                               |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| トリガー         | コミュニティ設定画面の「コミュニティを抜ける」ボタン                                                                               |
| 確認ダイアログ   | 「このコミュニティから退出しますか？\n・未来のアクティビティの参加は自動キャンセルされます\n・チャット履歴は閲覧できなくなります」 |
| バックエンド処理 | ① CommunityMembership 削除 → ② 未来のParticipation を自動キャンセル → ③ WaitlistEntry 削除 → ④ ChatChannel の DMParticipant 削除   |
| 制約             | OWNERは退出不可（先にオーナー移譲が必要）                                                                                          |

#### ③ ユーザー退会
| 項目             | 内容                                                                                                                                                                                       |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| トリガー         | マイページ「アカウントを削除」ボタン                                                                                                                                                       |
| 確認ダイアログ   | 「アカウントを削除しますか？この操作は取り消せません。\n・全コミュニティから退出します\n・オーナーのコミュニティがある場合は先に移譲してください\n・個人データは匿名化されます」           |
| バックエンド処理 | ① OWNER コミュニティ有無チェック（あればエラー） → ② 全 Membership の退出処理（②と同じ） → ③ User 論理削除（displayName → "退会済みユーザー"、email/avatarUrl → null） → ④ Credential 削除 |
| 制約             | OWNERコミュニティが残っている場合は退会不可                                                                                                                                                |

### 受入条件
- 3パターンそれぞれの確認ダイアログが正しく表示されること
- 影響範囲が明示されていること（箇条書き）
- 各バックエンド処理がトランザクション内で実行されること
- OWNERの退出/退会制約が正しく機能すること

### 関連ファイル（推定）
- `frontend/src/features/chat/` — チャット退出UI
- `frontend/src/features/community/` — コミュニティ退出UI
- `frontend/src/features/user/pages/MyPage.tsx` — 退会UI
- `backend/src/application/community/usecase/` — 退出UseCase
- `backend/src/application/user/usecase/` — 退会UseCase
- `backend/src/domains/community/` — 退出ドメインロジック

## W3-08 メンバー招待ボタン（LINE/メール/URL）

- **分類**: 新機能
- **優先度**: P2
- **変更対象**: 両方
- **変更レイヤー**: UI / API
- **由来**: 積み残し #6
- **依存**: なし

### 仕様
既存の `InviteToken` テーブルを活用し、3つの招待経路を提供するモーダルを実装する。

| 経路        | 動作                                                                                     |
| ----------- | ---------------------------------------------------------------------------------------- |
| **LINE**    | LIFF shareTargetPicker（LIFF環境）or LINEのURLスキーム（外部ブラウザ）で招待リンクを送信 |
| **メール**  | `mailto:` リンクで招待文面（URL含む）をプリセットして起動                                |
| **URL生成** | 招待URLを生成し、クリップボードにコピー                                                  |

- 招待URLフォーマット: `https://{domain}/invite/{token}`
- トークン有効期限: デフォルト7日（コミュニティ設定で変更可能にするかは Phase 2 スコープ外）
- 利用回数制限: デフォルト無制限（単一トークン複数回使用可）

### 受入条件
- コミュニティ詳細画面の招待ボタン押下でモーダルが表示されること
- 3経路（LINE/メール/URL）の選択肢が表示されること
- URL生成でクリップボードにコピーされ、トースト通知が出ること
- 招待URLからアクセスすると該当コミュニティへの参加フローに進むこと
- OWNER/ADMINのみ招待ボタンが表示されること

### 関連ファイル（推定）
- `frontend/src/features/community/` — 招待モーダルUI
- `backend/src/api/front/community/` — 招待トークン生成API
- `backend/src/application/community/usecase/` — InviteToken生成UseCase
- `backend/prisma/schema.prisma` — InviteToken テーブル（既存）

## W3-09 ビジター: 一括支払い更新UI

- **分類**: 機能拡張
- **優先度**: P2
- **変更対象**: 両方
- **変更レイヤー**: UI / API
- **由来**: wave2繰越
- **依存**: wave2 Phase4のビジター支払い管理（個別更新）が完了していること

### 仕様
管理者がスケジュール参加者一覧から複数のビジター（+通常メンバー）を選択し、支払い方法/ステータスを一括更新できるUIを実装する。

| 操作                 | 動作                                                                                  |
| -------------------- | ------------------------------------------------------------------------------------- |
| チェックボックス選択 | 参加者テーブルの各行にチェックボックスを追加。ヘッダーに全選択/全解除                 |
| 一括操作バー         | 選択件数表示 + 支払い方法ドロップダウン + 支払いステータスドロップダウン + 適用ボタン |
| 適用                 | 一括更新API呼び出し。成功後にテーブルを再取得                                         |

### 受入条件
- 参加者一覧テーブルにチェックボックス列が表示されること
- 1件以上選択すると一括操作バーが表示されること
- 一括適用後、選択した全員の支払い情報が更新されていること
- 部分失敗時はエラーの件数と対象者を表示すること

### 関連ファイル（推定）
- `frontend/src/features/activity/pages/ActivityDetailPage.tsx` — 参加者テーブル
- `backend/src/api/front/participation/` — 一括更新API（新設）
- `backend/src/application/participation/` — 一括更新UseCase

## W3-10 キャンセル待ち繰り上げ支払い（現金デフォルト自動作成）

- **分類**: 改善
- **優先度**: P2
- **変更対象**: 両方
- **変更レイヤー**: UseCase / Domain
- **由来**: 積み残し #8
- **依存**: なし

### 仕様
キャンセル待ちから繰り上げ参加になった場合、Paymentレコードを自動作成する。

| 項目                 | 内容                                                                                  |
| -------------------- | ------------------------------------------------------------------------------------- |
| 自動作成タイミング   | WaitlistEntry → Participation に昇格した瞬間                                          |
| デフォルト支払い方法 | `CASH`（現金）                                                                        |
| デフォルトステータス | `PENDING`（未済）                                                                     |
| 金額                 | Schedule.participationFee（ビジターの場合は Schedule.visitorFee ?? participationFee） |

**wave3スコープ**: 現金デフォルト自動作成のみ
**wave4送り**: PayPay/クレジット選択UI（#56 Stripe連携とまとめて対応）

### 受入条件
- キャンセル待ちから繰り上げ後、Payment レコードが自動作成されていること
- Payment.paymentMethod = CASH、Payment.status = PENDING であること
- 金額がスケジュールの参加費（ビジターの場合はビジター参加費）と一致すること
- アクティビティ詳細の支払い管理に繰り上げユーザーの行が表示されること

### 関連ファイル（推定）
- `backend/src/application/participation/usecase/` — キャンセル待ち繰り上げUseCase
- `backend/src/domains/participation/` — 繰り上げドメインロジック
- `backend/src/application/payment/` — Payment作成
