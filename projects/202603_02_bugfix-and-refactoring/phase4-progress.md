## 4-1: カレンダー D&D コピー — 設計メモ

### 要件
- **対象ユーザ**: コミュニティの OWNER / ADMIN のみ
- **操作**: アクティビティが存在する日付をドラッグ → 別の日付にドロップ
- **確認**: 「予定をコピーしますか？」ダイアログを表示
- **コピー内容**: 元のアクティビティ（Schedule）を複製。繰り返しルールはなし。日付のみ変更
- **コピー対象**: タイトル、説明、場所、参加費、上限人数、時間帯（開始/終了時刻はそのまま、日付だけ変更）

### 技術検討
- **ドラッグ元**: react-day-picker のカスタムDay コンポーネントで `draggable` 属性付与
- **ドロップ先**: 各日付セルに `onDrop` イベントハンドラ
- **ライブラリ**: `@dnd-kit/core` or HTML5 Drag and Drop API
  - `@dnd-kit/core` の方がモバイルタッチ対応が良い → 推奨
- **API**: `POST /v1/communities/:communityId/activities` を流用（コピー元データ + 新日付で新規作成）

### バックエンド変更
- 新規APIは不要（既存の作成APIを利用）
- フロント側でコピー元のアクティビティデータを取得し、日付を差し替えて POST

## 4-2: 返金オーケストレーション — 設計メモ

### 現状（Phase 3 時点）
- 支払い済みキャンセル時は確認ダイアログ表示 + 管理者通知
- 返金は手動（幹事⇔参加者間で直接やり取り）

### 将来設計（Phase 5 連携）
| 支払い方法 | 返金フロー                         | 実装状態                                                                |
| ---------- | ---------------------------------- | ----------------------------------------------------------------------- |
| STRIPE     | Stripe Refund API 自動呼び出し     | `StripePaymentStrategy` インターフェース定義済み、refund メソッド未実装 |
| CASH       | 管理者に「手動返金依頼」通知を送信 | Phase 3 の管理者通知を拡張                                              |
| PAYPAY     | PayPay Refund API or 手動返金通知  | 未設計                                                                  |

### 設計方針
1. `PaymentStrategy` インターフェースの `refund()` メソッドを各戦略クラスに実装
2. `CancelParticipationUseCase` 内で `paymentStatus === CONFIRMED` の場合に返金オーケストレーション呼び出し
3. Stripe Connect (Destination Charges) の返金: 手数料 3.6% の扱い検討（全額返金 or 手数料差引）
4. 返金ステータス管理: `PaymentStatus` に `REFUNDED` / `REFUND_PENDING` を追加

## 4-3: APIエラーハンドリング全体見直し — 設計メモ

### 背景
- BE で `PrismaClientValidationError` 等が発生しても FE 側は「アクティビティがありません」と空表示になるだけでエラーが見えない
- `useQuery` の `isError` を使っていない画面が多く、500系エラーが静かに握りつぶされている
- 開発時にもブラウザ上でエラーに気づけず原因調査に時間がかかる

### 検討項目
| #   | 項目                              | 概要                                                                                                              |
| --- | --------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| 1   | **グローバルエラートースト**      | `apiClient` (http) のレスポンスインターセプタで 4xx/5xx をキャッチし、トースト通知を表示                          |
| 2   | **useQuery エラーバウンダリ統一** | TanStack Query の `QueryCache.onError` でグローバルハンドリング。画面ごとの `isError` 分岐も標準テンプレート化    |
| 3   | **BE エラーレスポンス標準化**     | 全エラーレスポンスを `{ code, message, details? }` 形式に統一。Prisma エラーは catch → 適切な HTTP ステータス変換 |
| 4   | **開発時エラーオーバーレイ**      | `NODE_ENV=development` 時にスタックトレース付きオーバーレイ表示（Vite Error Overlay 活用）                        |
| 5   | **リトライ戦略**                  | 一時的な 5xx に対して TanStack Query の `retry` 設定を見直し（現状デフォルト3回）                                 |
| 6   | **フロントエラーログ収集**        | Sentry 等の導入検討。エラー発生率・頻度の可視化                                                                   |

### 前提条件
- Phase 5 の決済基盤整備（Stripe Connect 本番設定）が完了していること
- ビジネスポリシー: キャンセル期限（例: 開催24時間前まで全額返金）等の規約策定
