## PI-1: メッセージ全文検索の高速化

### 現状

- Backlog Phase 3（UBL-13）で `ILIKE` ベースのメッセージ検索を初期実装
- `SELECT ... WHERE content ILIKE '%keyword%'` はインデックスが効かず、テーブルフルスキャンになる

### 移行候補

| 方式                             | メリット                                                                          | デメリット                                                   | 導入コスト                                          |
| -------------------------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------------ | --------------------------------------------------- |
| **pg_trgm 拡張**                 | PostgreSQL ネイティブ。GIN/GiST インデックスで `ILIKE` / `similarity()` が高速化  | 日本語対応が弱い（トライグラム分割が英語前提）。部分一致のみ | 低（`CREATE EXTENSION pg_trgm` + インデックス追加） |
| **PGroonga**                     | PostgreSQL 拡張。日本語全文検索に強い（MeCab 形態素解析対応）                     | RDS では使えない（EC2 or 自前 DB が必要）                    | 中（拡張インストール + インデックス設計）           |
| **Elasticsearch / OpenSearch**   | 大規模データに最適。日本語 Analyzer（kuromoji）完備。曖昧検索・関連度スコアリング | 別サービスの運用コスト。データ同期の仕組みが必要             | 高（インフラ追加 + 同期パイプライン）               |
| **Amazon OpenSearch Serverless** | マネージドで運用不要。日本語対応                                                  | コストが読みにくい。ロックイン                               | 中〜高                                              |

### 判断基準

- メッセージ数が **10万件未満**: `ILIKE` で十分（インデックスなしでも数百ms以内）
- メッセージ数が **10万〜100万件**: `pg_trgm` + GIN インデックスで対応
- メッセージ数が **100万件超** or **日本語精度重視**: Elasticsearch / OpenSearch を検討

### 移行タイミング

Phase 3 完了後、実際のメッセージ量を計測してから判断。当面は `ILIKE` で運用。
